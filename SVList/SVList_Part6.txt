Part6 ObjectPoolManager（多模板对象池系统）
6.1 为什么 Queue<GameObject> 不够？

最简单：

Queue<GameObject> pool;

只能支持：

ItemPrefab

但实际：

聊天：

TextMessagePrefab

ImageMessagePrefab

GiftMessagePrefab

VoiceMessagePrefab

商城：

WeaponPrefab

HeroPrefab

SkinPrefab

不同类型不能混。

否则：

ImageMessage

可能被拿去显示：

TextMessage。

直接炸。

所以：

对象池必须：

一个 Prefab 一个池

6.3 PrefabID

不要：

Dictionary<GameObject,PoolNode>

因为：

GameObject引用查找较重。

推荐：

初始化：

TextPrefab -> 0

ImagePrefab -> 1

GiftPrefab -> 2

保存：

int prefabID

整个系统：

都使用 int。

包括：

ActiveItem

CreateRequest

Renderer

全部使用：

PrefabID

提高效率。

6.4 PoolNode

真正的小池。

class PoolNode
{
    int PrefabID;

    GameObject Prefab;

    Queue<ISVItemRenderer> Pool;

    int CurrentCount;

    int MaxCount;
}

职责：

只管理一种 prefab。

6.5 Get()

流程：

先：

if(queue.Count>0)

6.6 Release()

流程：

检查：

CurrentCount<MaxCount

成立：

进入：

Queue。

否则：

真正 Destroy。

避免：

池无限增长。

6.9 PoolRoot是否SetActive(false)

很多人喜欢：

poolRoot.SetActive(false)

错。

因为：

子节点：

OnDisable

OnEnable

反复触发。

推荐：

保持：

poolRoot active

通过：

gameObject.SetActive(false)

控制单个 Item。

6.10 Item隐藏方式

推荐：

renderer.gameObject.SetActive(false)

而不是：

scale=0

或者：

CanvasGroup alpha=0

因为：

仍然参与：

Layout。

浪费性能。

6.22 AI最容易写错
错误1

一个 Queue 管所有 Prefab。

MultiTemplate崩。

错误2

不限制 MaxPool。

池无限长。

错误3

PoolRoot inactive。

触发大量：

OnEnable。

错误4

Destroy 后不减 Count。

统计错。

错误5

Addressables 使用 Destroy。

资源泄漏。

错误6

GetComponent 每次 Get。

应该：

OnCreate缓存。

错误7

new Queue()

产生 GC。

错误8

Recycle 后立即 Create。

不断：

SetParent。

Canvas rebuild 风暴。
