2.1 设计原则

整个 SVList 的性能几乎完全由数据结构决定。

目标：

0GC
O(logN)
百万数据支持
最小重建
最小查询
最小拷贝

原则：

尽量避免
List.Remove()
Dictionary.Values
LINQ
foreach

因为：

可能产生GC
O(N)

推荐：

Dictionary<int,Renderer>
Queue<T>
数组
2.2 SVListState

整个列表运行状态。

生命周期
List存在期间唯一
类定义
class SVListState
{
    // 当前滚动距离
    float ScrollOffset;

    // 当前Content总长度
    float ContentSize;

    // 当前可见范围
    int FirstVisibleIndex;

    int LastVisibleIndex;

    // 上一帧范围
    int PrevFirstVisibleIndex;

    int PrevLastVisibleIndex;

    // Viewport尺寸
    float ViewportSize;

    // 是否需要刷新
    bool Dirty;

    // 当前状态
    SVListStatus Status;
}
AI容易漏：

PrevFirst / PrevLast

没有上一帧范围：

无法Diff。

只能全刷新。

2.3 VisibleRange

表示可见区。

struct VisibleRange
{
    public int First;

    public int Last;
}

注意：

用 struct。

原因：

生命周期短。

避免：

new VisibleRange()

产生GC。

支持：

bool Equals()

用于：

if(oldRange==newRange)
return;

避免无意义刷新。

2.4 HeightInfo

动态高度核心。

class HeightInfo
{
    // 是否已经真实测量
    bool IsMeasured;

    // 当前高度
    float Height;

    // 默认估计高度
    float EstimateHeight;
}

生命周期：

数据量大小

100万数据：

100万个 HeightInfo。

这里 AI 很容易犯错。

直接：

float[] heights

不够。

因为：

有些 item：

还没生成。

高度未知。

不能：

0

否则：

BinarySearch错。

必须：

区分：

未测量
EstimateHeight
已测量
真实Height

2.6 ItemState
enum ItemState
{
    Pooled,

    Active,

    PendingRecycle

为什么有 PendingRecycle？

AI最喜欢：

直接：

放回池。

但是：

此时：

可能还在 Layout rebuild。

会炸。

所以：

先：

PendingRecycle

求：

对应哪个Index。

AI最容易遗漏的问题
① 重复回收

解决：

ItemState。

② 没有 PrevRange

导致全刷新。

③ ActiveItem 不保存 PrefabID

MultiTemplate崩。

④ Height=0

BinarySearch错。

必须：

EstimateHeight。

⑤ List.RemoveAt(0)

O(N)

应该 Queue。

删除元素错乱。

应该保存：

currentIndex

数据实时向 DataSource 要。

⑦ VisibleRange 用 class

大量GC。

应该 struct。
